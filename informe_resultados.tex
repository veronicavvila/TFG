% ===================== INTRO =====================
\section{Contexto}

El pipeline evaluado es:
\begin{enumerate}
    \item Entrenar un clasificador $P \text{ vs } U$ (regresión logística) para estimar $P(S=1 \mid x)$.
    \item Estimar $\hat{\alpha}$ a partir de esos scores.
    \item Corregir las probabilidades para obtener $\hat{P}(Y=1 \mid x)$ y calcular la MI corregida (\textbf{MI-PU}).
    \item Comparar el ranking resultante con el ranking ``naive'' (MI calculada directamente sobre $S$ en vez de $Y$) y con el ranking real (calculado con las etiquetas $Y$ verdaderas).
\end{enumerate}

La calidad del ranking se mide con la \textbf{correlación de Spearman} entre el ranking estimado y el real. Se varía $\alpha \in \{0.05, 0.1, 0.2, 0.3, 0.5\}$ con 5 semillas diferentes.

\medskip
Los resultados esperados que se quieren validar son:
\begin{enumerate}
    \item[\textbf{H1}] \textbf{MI naive falla}: confundir positivos ocultos con negativos degrada el ranking.
    \item[\textbf{H2}] \textbf{PU mejora}: la corrección propuesta debe aproximarse mejor al ranking real.
    \item[\textbf{H3}] \textbf{$\hat{\alpha}$ converge}: la estimación de $\alpha$ debe acercarse al valor verdadero.
\end{enumerate}

Se probaron ambas variantes con y sin ruido. \textbf{Sin ruido} sirve como \textit{baseline} controlado para verificar el comportamiento esperado. \textbf{Con ruido} simula un escenario más realista donde la calidad de las medidas clínicas puede estar degradada, algo habitual cuando las imágenes tienen artefactos o el proceso de extracción de features no es perfecto.

\subsection{Análisis}

\textbf{Sin ruido} -- las tres hipótesis de resultados esperados se cumplen:
\begin{itemize}
    \item \textbf{H1}: el naive falla, especialmente a $\alpha$ bajos. Para $\alpha = 0.05$ la correlación naive es apenas $0.44$, confirmando que con pocos positivos etiquetados el ranking naive es pobre.
    \item \textbf{H2}: el método PU mejora claramente ($0.76$ para $\alpha=0.05$, subiendo hasta $0.97$ para $\alpha=0.5$). La diferencia con el naive es muy grande con valores bajos de $\alpha$, que es exactamente el caso esperado.
    \item \textbf{H3}: la estimación es muy cercana al valor real ($0.493$ vs $0.5$ para $\alpha=0.5$). Para $\alpha=0.05$ hay una sobreestimación de $\approx 0.04$, hay muy pocos positivos etiquetados y el clasificador tiene dificultad para calibrar los scores.
\end{itemize}

\textbf{Con ruido} -- la tendencia se mantiene pero con valores absolutos más bajos:
\begin{itemize}
    \item El ruido aumenta las estimaciones de $\hat{\alpha}$ para valores pequeños (estima $0.12$ cuando el real es $0.05$), lo que significa que la señal PU se confunde con el ruido.
    \item La correlación naive baja aún más ($0.21$ para $\alpha=0.05$), mientras que PU se mantiene por encima de $0.44$.
    \item El método PU sigue siendo claramente superior al naive. La robustez al ruido es razonable.
\end{itemize}

Breast Cancer es el dataset donde el método funciona de forma más limpia y donde se validan las tres hipótesis esperadas.

\subsubsection{Método robusto: positivos confiables}

Se evaluó la estimación robusta de $\alpha$ (top 20\%) en Breast Cancer sin ruido y variando el percentil desde 5\% hasta 100\%. Los resultados muestran un patrón significativo: aunque el método robusto \textbf{sobrestima severamente} la estimación de $\hat{\alpha}$, este error \textbf{NO degrada los resultados finales} (AUC, Spearman) del ranking de features.

\paragraph{Estimación de $\alpha$:} El método robusto empeora la estimación en todo el rango:
\begin{itemize}
    \item Para $\alpha = 0.05$: $\hat{\alpha}_{\text{media}} = 0.086$ (error 72\%) vs $\hat{\alpha}_{\text{robusto}} = 0.179$ (error 258\%)
    \item Para $\alpha = 0.30$: $\hat{\alpha}_{\text{media}} = 0.315$ (error 5\%) vs $\hat{\alpha}_{\text{robusto}} = 0.490$ (error 63\%)
    \item Para $\alpha = 0.50$: $\hat{\alpha}_{\text{media}} = 0.493$ (error 1\%) vs $\hat{\alpha}_{\text{robusto}} = 0.699$ (error 40\%)
\end{itemize}

Los ejemplos con scores más altos (top 20\%) no son representativos de la proporción real $\alpha$, lo que explica la sobreestimación persistente.

\paragraph{Variación de porcentajes:} A pesar de variaciones de alpha del $180\%$ (desde $\hat{\alpha}=0.23$ hasta $\hat{\alpha}=0.57$ con percentiles extremos [5\%, 100\%]), las métricas finales del ranking permanecen \textbf{prácticamente invariables}:

\begin{itemize}
    \item \textbf{AUC}: $0.9805 \to 0.9800$ (variación de apenas $\mathbf{0.05\%}$, prácticamente flat)
    \item \textbf{Spearman}: varía moderadamente en rango $[0.035, 0.080]$ pero en escala muy pequeña
    \item \textbf{Top-10 features}: idénticas independientemente del percentil utilizado
\end{itemize}

\paragraph{Conclusión:} Este comportamiento revela que \textbf{Breast Cancer es un dataset donde el método es robusto a errores en la estimación de $\alpha$}. Aunque el método robusto falla a nivel de estimación puntual de $\alpha$, el ranking final de features es insensible a esa degradación. Esto sucede porque:

\begin{enumerate}
    \item Las features informativas de Breast Cancer son tan claras y bien separadas que pequeñas variaciones en $\hat{\alpha}$ (que alteran ligeramente las probabilidades estimadas $\hat{P}(Y=1 \mid x)$) no cambian el orden relativo en el ranking MI.
    \item El dataset es suficientemente limpio y balanceado que la corrección PU converge a un espacio de solución estable: múltiples valores de $\alpha$ producen prácticamente el mismo ranking final porque todos logran separar bien los positivos verdaderos de los negativos.
    \item Este es un \textbf{resultado positivo}: demuestra que en datasets "fáciles", el método PU no es sensible a errores de calibración de $\alpha$.
\end{enumerate}

En resumen, aunque el ranking final de features es insensible a errores de calibración de $\alpha$ (gracias a la claridad de las features), el AUC no mejora estadísticamente con el método robusto. Recomendación: usar mean en lugar de robusto en Breast Cancer para simplificar la metodología sin perder desempeño.

\section{Sonar}

\subsection{Características del dataset}

Señales sonar para distinguir rocas de minas submarinas (\textit{Sonar, Mines vs Rocks}). Son \textbf{208 muestras} con \textbf{60 features} (energías de la señal en diferentes bandas de frecuencia). Es un dataset binario perfectamente balanceado ($50\%$ / $50\%$), pero difícil por la alta relación features/muestras ($60/208 \approx 0.29$).

Con tan pocas muestras, añadir ruido gaussiano extra haría los estimadores de MI demasiado complejos. Ya en bruto los resultados son muy ruidosos; el experimento con ruido no aportaría información útil y dificultaría la interpretación.

\subsection{Análisis}

\begin{itemize}
    \item \textbf{H1}: el naive sí falla ($0.05$--$0.20$). El ranking naive no contiene información útil sobre el ranking real.
    \item \textbf{H2}: el PU tampoco funciona ($0.007$--$0.15$). La corrección PU no logra mejorar el ranking, sino que en algún caso lo empeora. No tiene sentido corregir con un $\hat{\alpha}$ erroneo.
    \item \textbf{H3}: la estimación de $\hat{\alpha}$ falla completamente. Independientemente del valor real, el estimador devuelve valores en torno a $0.40$--$0.55$, que corresponde aproximadamente a la proporción de positivos en el dataset ($50\%$). El clasificador logístico está sobreajustando: con $\approx 3.5$ muestras por feature, no puede aprender, y los scores $P(S=1 \mid x)$ son muy aleatorios.
\end{itemize}

El fallo lo causa el ratio muestras/features. Con tan poca información, todo el pipeline se degrada. Este dataset requeriría reducción de dimensionalidad previa (p.ej.\ PCA) o un clasificador más regularizado para que el método tuviera alguna posibilidad de funcionar.

\subsubsection{Método robusto: positivos confiables}

Se evaluó la estimación robusta de $\alpha$ en Sonar variando el percentil desde 5\% hasta 100\% (con $\alpha_{\text{true}}=0.2$). Los resultados muestran un patrón \textbf{catastrófico}: la sobreestimación es extrema en todo el rango y no mejora significativamente incluso con top 100\%.

\paragraph{Estimación de $\alpha$:} El método robusto produce errores devastadores:
\begin{itemize}
    \item Top 5\%: $\hat{\alpha} \approx 0.92$ (error $\approx 360\%$) — sobrestimación extrema
    \item Top 10\%: $\hat{\alpha} \approx 0.89$ (error $\approx 345\%$)
    \item Top 20\%: $\hat{\alpha} \approx 0.83$ (error $\approx 300\%$)
    \item Top 50\%: $\hat{\alpha} \approx 0.71$ (error $\approx 255\%$)
    \item Top 100\%: $\hat{\alpha} \approx 0.53$ (error $\approx 165\%$) — aún extremadamente lejano
\end{itemize}

Incluso usando el 100\% de datos (es decir, toda la información disponible), el estimador sobrestima por más de $3.3\times$ el valor real. Esto es el indicador más claro de que el problema es \textbf{estructural}, no metodológico.

\paragraph{Impacto en métricas finales, invariabilidad por colapso:} Contraintuitivamente, AUC se mantiene prácticamente constante:

\begin{itemize}
    \item \textbf{AUC}: varía en rango $[0.759, 0.764]$ (variación de $\sim 0.6\%$), prácticamente flat
    \item \textbf{Spearman}: muy bajo en todo rango $[0.05, 0.21]$, sin patrón claro
    \item \textbf{Ranking}: completamente inestable, reflejando la falta total de señal PU
\end{itemize}

La invariabilidad de AUC no es un aspecto positivo: ocurre porque el clasificador PU ha colapsado y los scores son esencialmente aleatorios. Todas las estimaciones de $\alpha$ son igualmente malas, por lo que el comportamiento es uniforme.

\paragraph{Conclusión, inaplicable categóricamente en Sonar:} Sonar es un caso extremo que demuestra los límites absolutos del método PU:

\begin{enumerate}
    \item El dataset tiene ratio muestras/features de apenas $\approx 3.5:1$ (208 muestras / 60 features). Esta es una proporción donde incluso métodos clásicos de aprendizaje supervisado fallan. El clasificador logístico no puede aprender una frontera confiable de separación, y los scores resultan esencialmente aleatorios.
    \item El método robusto amplifica el ruido: al seleccionar el top 20\% o restringirse a percentiles bajos, limita aún más la información disponible, haciendo los estimadores aún más inestables.
    \item La persistencia del error incluso a top 100\% (165\% de error residual) sugiere que no hay ningún subconjunto de ejemplos que sea suficiente para representar $\alpha$ en este dataset.
    \item El Spearman bajo (máximo $\approx 0.21$) confirma que ningún ranking derivado del método PU tiene valor predictivo.
\end{enumerate}

En resumen, Sonar es problemático para PU, pero el método robusto logra una mejora estadística marginal: test Robust > Mean: $p=0.0166$ significativo. Aunque el rendimiento absoluto es pobre (AUC $\approx 0.697$, Spearman $\approx 0.21$), el robusto obtiene una mejora de sobre mean (diferencia AUC: +0.009).

Recomendación: Si se insiste en usar Sonar, preferir método robusto sobre mean, aunque ambos son inaplicables. El dataset requeriría:

\begin{itemize}
    \item Reducción de dimensionalidad (PCA, selección de features previos)
    \item Un modelo de clasificación más regularizado
    \item O simplemente reconocer que los 208 puntos disponibles son insuficientes para 60 features
\end{itemize}

Sin estos cambios, no se recomienda usar el método PU en Sonar bajo ninguna variante.

\subsubsection{Método robusto: positivos confiables}

\paragraph{Estimación de $\alpha$:} El método robusto empeora la estimación y converge lentamente:
\begin{itemize}
    \item Top 5\%: $\hat{\alpha} \approx 0.56$ (error $\approx 178\%$) — sobrestimación severa
    \item Top 10\%: $\hat{\alpha} \approx 0.51$ (error $\approx 150\%$)
    \item Top 20\%: $\hat{\alpha} \approx 0.46$ (error $\approx 130\%$)
    \item Top 50\%: $\hat{\alpha} \approx 0.35$ (error $\approx 70\%$)
    \item Top 100\%: $\hat{\alpha} \approx 0.25$ (error $\approx 25\%$) — aún lejano del valor real
\end{itemize}

La convergencia es lenta; incluso con top 100\% el error sigue siendo del 25\%.

\paragraph{Degradación clara en métricas:} A diferencia de Breast Cancer y Spambase, Ionosphere muestra sensibilidad negativa al método robusto:

\begin{itemize}
    \item \textbf{AUC}: varía en rango $[0.821, 0.836]$ (variación de $\sim 1.8\%$), considerada pequeña pero presente
    \item \textbf{Spearman}: muestra variabilidad muy baja en rango $[0.005, 0.195]$, con picos alrededor de 10-20\%
    \item \textbf{Estabilidad de características}: el ranking es poco estable, reflejo de que los scores estimados son poco confiables
\end{itemize}

\paragraph{Conclusión:} Ionosphere demuestra que el método robusto \textbf{no es recomendable} bajo estas condiciones:

\begin{enumerate}
    \item El dataset es pequeño (351 muestras para 34 features, ratio $\approx 10:1$) con features fuertemente correlacionadas. Los scores del clasificador PU no son confiables, y seleccionar arbitrariamente el top 20\% amplifica el ruido.
    \item La lenta convergencia conforme aumenta el percentil (aún 25\% de error a top 100\%) sugiere que los positivos etiquetados están distribuidos de manera no uniforme en el espacio de scores, haciendo que ningún percentil anterior al 100\% sea representativo.
    \item El Spearman bajo sugiere que independientemente del percentil, el ranking es frágil y poco informativo del ranking real.
\end{enumerate}

En resumen, Ionosphere es un caso donde el método robusto es \textbf{contraproducente}. A diferencia de spambase donde la robustez es aplicable con percentiles altos, en Ionosphere la estructura inherente del dataset (pequeño, correlacionado, con señal débil) hace que el método robusto no pueda mejorar la estimación de $\alpha$. No se recomienda usar estimación robusta en este dataset.

\textbf{Gas Sensor Drift es un dataset multiclase, no binario}, y este es el problema que explica todos los resultados anómalos que se observan.

El dataset original contiene \textbf{6 clases de gas} (Ethanol, Ethylene, Ammonia, Acetaldehyde, Acetone, Toluene). Para poder aplicar el framework PU se elige una de las 6 clases como ``positivo'' y se etiquetan todas las demás como ``negativo''. Esto genera un conjunto de negativos \textbf{heterogéneo}: no son muestras de una clase coherente, sino la unión de 5 tipos de gas completamente diferentes entre sí, cada uno con su propia distribución en el espacio de features.

Este diseño rompe un supuesto fundamental del método PU: que los negativos (unlabeled que son realmente negativos) provienen de una distribución homogénea y bien definida. En la práctica ocurre lo siguiente:

\begin{itemize}
    \item El clasificador $P$ vs $U$ no aprende a separar ``este gas'' de ``no este gas'', sino que intenta construir una frontera entre una clase compacta y una mezcla de 5 distribuciones muy distintas. Aunque la mezcla de negativos cubre regiones del espacio de features muy alejadas de los positivos, haciendo la separación $P$ vs $U$ trivialmente lineal y con altos scores incluso sin corrección PU.
    \item Como consecuencia, los scores $\hat{P}(S=1 \mid x)$ capturan principalmente ``pertenencia a la clase del positivo'' de forma casi perfecta, lo que explica que $\hat{\alpha}$ converja bien (H3). Pero esta separación: no se debe a que el método PU funcione, sino a que los negativos heterogéneos son fácilmente distinguibles en nivel de features.
    \item El ranking de features refleja qué sensores discriminan mejor entre las 6 clases de gas globalmente, no qué sensores son informativos desde la perspectiva binaria positivo/negativo. Por eso el naive, que usa directamente las etiquetas binarizadas, ya captura ese ranking correctamente sin necesitar ninguna corrección.
\end{itemize}

En un dataset verdaderamente binario, los positivos ocultos en el unlabeled se confunden con los negativos y degradan el ranking naive (H1). Aquí eso no ocurre porque la señal discriminativa de los sensores es tan fuerte y la separación entre las 6 clases tan clara que el ruido PU queda enmascarado.

\textbf{Conclusión:} los resultados de Gas Sensor no invalidan el método propuesto: el método asume negativos aproximadamente i.i.d.\ de una única distribución. Cuando los negativos son una mezcla de múltiples clases heterogéneas, el escenario PU deja de ser el que el método estudia.

\subsection{Análisis de características críticas}

\subsubsection{Casos de ÉXITO: Phoneme y MAGIC Telescope}

\textbf{Características comunes:}
\begin{enumerate}
    \item \textbf{Balance de clases moderado}: Ambos datasets tienen positivos entre 35-50\%:
    \begin{itemize}
        \item Hay suficientes positivos etiquetados para que el clasificador PU aprenda
        \item Hay suficientes negativos genuinos que se confunden y causan degradación observable
        \item Los top 20-30\% de scores son representativos de la verdadera proporción $\alpha$
    \end{itemize}
    
    \item \textbf{Tamaño de muestra grande}: $n \geq 5{,}000$ muestras
    \begin{itemize}
        \item Phoneme: 5{,}404 muestras, ratio n/features = 1{,}081:1
        \item MAGIC Telescope: 19{,}020 muestras, ratio n/features = 1{,}902:1
        \item Garantiza suficiente variabilidad estadística para estimar $\hat{\alpha}$ con precisión
    \end{itemize}
    
    \item \textbf{Features moderadamente discriminativas}: No triviales, pero no demasiado difíciles
    \begin{itemize}
        \item Phoneme: 5 features acústicas, correlación moderada
        \item MAGIC Telescope: 10 features astronómicas bien seleccionadas
        \item Permiten que el método PU capture diferencias reales entre positivos y negativos
    \end{itemize}
\end{enumerate}

\textbf{Resultados observados:}
\begin{itemize}
    \item \textbf{Phoneme}: Convergencia casi perfecta ($\hat{\alpha}$ error 1-16\% con óptimo de 1\% a top 30\%). AUC invariable, Spearman 0.88-0.96. Este es el caso paradigmático.
    \item \textbf{MAGIC Telescope}: Mejora de 23\% a 2-9\% en error de $\hat{\alpha}$. Ranking estable. Demuestra que incluso con factor error de 23\%, el método robusto logra casi perfección.
\end{itemize}

\subsubsection{Caso INTERMEDIO: Spambase}

\textbf{Características:}
\begin{itemize}
    \item Balance: 39\% positivos (dentro del rango óptimo)
    \item N muestras: 4{,}601 (en el límite inferior de viabilidad)
    \item Ratio n/features: 81:1 (moderado, comparado con casos exitosos)
    \item Features: 57 features de texto ya procesadas
\end{itemize}

\textbf{Resultado especial:}
El método robusto funciona \textbf{dependiendo del percentil elegido}. Error de $\hat{\alpha}$:
\begin{itemize}
    \item Top 5-20\%: Sobrestima severamente (100-145\% error)
    \item Top 50-100\%: Converge razonablemente (10-50\% error)
\end{itemize}

\textbf{Conclusión}: Spambase es aplicable pero \textbf{requiere cuidado en selección del percentil}. La práctica recomendada es usar percentiles altos ($\geq 50\%$) para minimizar sesgo. No es un fracaso categórico como Sonar, pero tampoco un éxito automático como Phoneme.

\subsubsection{Casos de FRACASO: Características comunes}

Los cuatro casos de fracaso comparten patrones distintos pero igualmente problemáticos:

\paragraph{1. Ionosphere y Sonar: Insuficiencia de datos}


\textbf{Ionosphere:}
\begin{itemize}
    \item Balance: 64\% positivos (muy sesgo hacia positivos)
    \item N: 351 muestras, features: 34 → ratio 10.3:1
    \item Problema: Pequeño con features correlacionadas
    \item Resultado: Convergencia lenta, error aún 25\% a top 100\%
\end{itemize}

\textbf{Sonar:}
\begin{itemize}
    \item Balance: 50\% (aparentemente ideal)
    \item N: 208 muestras, features: 60 → ratio 3.5:1 (CRÍTICO)
    \item Problema: Muchas features por muestra
    \item Resultado: Error 165\% incluso con top 100\%, todo el pipeline colapsa
\end{itemize}

\textbf{Diagnóstico común}: El clasificador logístico no puede aprender. Los scores $P(S=1 \mid x)$ son esencialmente aleatorios. El método robusto amplifica este ruido en lugar de mejorarlo.

\textbf{¿Por qué falla el método robusto aquí?}

- Seleccionar ``los mejores'' ejemplos de un pool de scores aleatorios sigue siendo principalmente ruido

- No es un fallo del método, sino del dataset que viola la estructura (n >> p)

\paragraph{2. MiniBooNE: Limitación conceptual severa}


\textbf{Características:}
\begin{itemize}
    \item Balance: 72\% positivos (clase mayoritaria)
    \item N: 130{,}065 muestras (enorme)
    \item Features: 50 → ratio 2{,}601:1 (excelente en teoría)
\end{itemize}

\textbf{Resultado:}
El método robusto logra \textbf{convergencia técnica casi perfecta} ($\hat{\alpha}$ error 6\% a top 30\%) pero el Spearman ranking no mejora (se mantiene ~0.05). 

\textbf{Diagnóstico}: No es un problema de estimación de $\alpha$. Es una \textbf{limitación conceptual del framework PU}:
\begin{itemize}
    \item Con 72\% de positivos verdaderos, los ejemplos etiquetados $S$ (subset de positivos) casi no introduce confusión
    \item El método PU ``intenta corregir'' una degradación que no existe realmente
    \item El ranking naive ya es casi óptimo (Spearman 0.97-0.99)
    \item Mejorar $\hat{\alpha}$ no rescata el método porque el problema no es $\alpha$
\end{itemize}

\textbf{Conclusión}: No es una limitación de robustez, sino de los \textbf{supuestos fundamentales del framework PU}, que asume balance de clases moderado.

    \item \textbf{Sonar}: $p = 0.0166$ (mejora marginal, error 0.0089 AUC)
\end{itemize}

\subsubsection{ROBUST NO mejora MEAN (p>0.05)}

\subsection{Conclusiones de validación: Cuándo y dónde funciona el método}

\subsubsection{Condición 1: Method robustness es crítico cuando error MEAN alto (>15\%)}

Datasets donde se observa error de $\hat{\alpha} > 15\%$ con método mean:
\begin{itemize}
    \item MAGIC Telescope (mean error 23\% → robust error 4\%)
    \item Phoneme (mean error 22\% → robust error 4-10\%)
    \item Sonar (mean error 98-956\% → robust semi-falla)
    \item Ionosphere (mean error 3-86\% → robust empeora)
\end{itemize}

\textbf{En estos casos}: Usar ROBUST (p<0.05 de mejora en primeros 3, o por diagnóstico de insuficiencia de datos en Sonar).

\subsubsection{Condición 2: Balance de clases determina viabilidad conceptual}

\begin{itemize}
    \item \textbf{30-50\% positivos}: PU funciona bien. Sonar (50\%) falla por ratio n/features, no por balance.
    \item \textbf{62-72\% positivos}: PU no funciona conceptualmente. MiniBooNE y Ionosphere fallan incluso con datos adecuados.
\end{itemize}

\subsubsection{Condición 3: Tamaño de muestra}

\begin{itemize}
    \item $n \geq 5{,}000$ (Phoneme, MAGIC, Spambase): PU funciona, robusto ayuda cuando mean error alto
    \item $n = 3{,}500$-$5{,}000$ (Spambase en borde): Funciona pero marginal
    \item $n < 1{,}000$ (Ionosphere 351, Sonar 208): Falla o muy arriesgado
\end{itemize}

\subsection{Diagnóstico y Conclusiones para Microarrays}

\textbf{Sobreajuste inevitable}: Cualquier clasificador (PU o no) tiende a ajustarse al ruido más que a patrones reales. El ranking que produce no refleja relaciones verdaderas entre features.

{Conclusión sobre métodos robusto vs mean:} En microarrays, ambos métodos fallan por la misma razón subyacente (sobre-parametrización), no porque uno sea mejor que otro. El hecho de que robust NO mejore mean es indicativo de que el problema no es estimación de $\alpha$, sino viabilidad fundamental del enfoque.

Los datasets de microarrays son idadecuados para el método PU en selección de features. La razón no es un defecto del método PU, sino un desajuste fundamental entre la metodología (requiere n $\gg$ p para inferencia estadística confiable) y la naturaleza de los datos (n $\ll$ p). 

En este escenario, usar MI-PU produce rankings indistinguibles de usar MI-Naive (o incluso aleatorio). La información contenida en $S$ no es suficiente para que el paso de corrección por $\alpha$ contribuya valor. Se necesita o bien:
\begin{itemize}
    \item Muestras mucho más grandes ($n \geq 1{,}000$), o
    \item Reducción de dimensionalidad previa ($p \approx 50$-$100$), o
\end{itemize}


\subsubsection{Recomendaciones Operativas}

\begin{enumerate}
    \item \textbf{No usar PU en microarrays sin preprocesamiento} (p.ej., feature selection previo o PCA a 50-100 componentes)
    
    \item \textbf{Si se insiste en usar PU}:
    \begin{itemize}
        \item Aplicar regularización fuerte (L2) en el clasificador PU
        \item Usar validación cruzada estricta (leave-one-out en datasets tan pequeños)
        \item Considerar métodos de ensemble para reducir varianza
    \end{itemize}
    
    \item \textbf{Alternativas recomendadas}:
    \begin{itemize}
        \item Métodos de test estadísticos simples (t-test, Mann-Whitney) con corrección Bonferroni
        \item Métodos bayesianos diseñados para high-dimensional, low-n (prior informativo)
        \item Métodos de RNA-seq específicos del dominio (limma, DESeq2) si los datos lo permiten
    \end{itemize}
\end{enumerate}

    \item \textbf{Criterios compartidos}: $n > 5{,}000$, balance de clases 30-72\%, ratio n/p $> 80:1$
\end{itemize}

\subsubsection{Datasets de INTERMEDIO (Preferir Mean, Robusto opcional)}
\begin{itemize}
    \item \textbf{Spambase, Breast Cancer}: Usar mean como primera opción. Robusto no proporciona mejora estadística.
    \item \textbf{Ionosphere}: Usar mean. Robusto empeora significativamente (p=0.9998).
    \item \textbf{Criterios compartidos}: $n < 5{,}000$ o balance desbalanceado (>60\%)
\end{itemize}

\subsubsection{Datasets de FRACASO (Evitar)}
\begin{itemize}
    \item \textbf{Sonar}: Ratio n/p extremadamente bajo (3.5:1). Ni mean ni robust funcionan. Si es imprescindible, usar mean como fallback.
    \item \textbf{Microarrays (CNS, Colon, DLBCL)}: n $\ll$ p hace ambos métodos inútiles. Requieren preprocesamiento (PCA, feature selection previo) antes de aplicar PU.
\end{itemize}
